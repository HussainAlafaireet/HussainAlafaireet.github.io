Physics CSS Browser

Controlling AR with Everyday Web Tech


View Process   View Final Output

Project Overview

How can we help web programmers control Augmented Reality the same way they control web pages?

Roles: UX Researcher, UX Designer  |  Duration: 4 Months
Tools Used: AFrame Web AR Framework, Adobe Photoshop/Illustrator
UX Skills Used: Design Sketching, AR/VR Prototyping, UX Research Review

Summary: I was an academic UX researcher for a project sponsored by Mozilla and the University of Michigan. The aim of the project was to develop visions and future directions for the concept of a Mixed Reality internet browser - a browser that can intermix traditional 2-D Internet content with Augmented and Virtual Reality content.

I provided an answer to one question within this vision: How can we adapt everyday web technology to control the look, behavior, and position of AR content intelligently?


My Contributions:

I developed the "Physics CSS Browser" model - a vision of controlling AR objects using everyday web technology. This vision is built on 4 distinct design elements:

  1. The AR Container: a flexible container for AR content
  2. The Room Model:: a model for AR content placement that makes intuitive sense
  3. The Physics Playground: An exploration area for controlling AR content placement with game physics
  4. The Physic CSS Browser: The model that ties the design elements together.

Process

I worked as an academic researcher on a project sponsored by Mozilla the University of Michigan. The goal of the project was to explore design concepts for a mixed reality browser - an internet browser designed to merge normal web content with Augmented Reality (AR) content.

My specific goal was to develop a method by which web technology could be used to control the size and positioning of AR objects. Achieving this goal would allow anyone with web programming experience to use AR content in their webpages. The result of my work was the Physics CSS Browser model..

Brainstorming

Project Research Questions

  • What existing web technologies can we extend to control AR content in a similar manner to 2-D content?
  • How can we turn a real-world location into a usable "AR webpage"?
  • How can we position "AR webpage" content into the real-world in a way that's intuitive and adaptable?
  • Can we use leverage simulated physics to automatically control AR content intuitively?

The AR Container

I began the project by brainstorming ways basic web technology might be used to control AR/VR content.

Cascading Style Sheets (CSS) immediately jumped to mind as a control method for AR content. CSS is the current way web programmers control how their websites look. Colors, fonts, sizing, and positioning can all be handled in CSS. This CSS is then applied to all webpages in a given website, to create a consistent look and feel.

The AR Container



To extend the Box Model to cover AR content. I developed the concept of the AR Container. The AR Container is a 3-D container that contains and controls AR content positioned in the real-world.

Room Model

Now that a vision for control of an individual piece of AR content had been laid out, it was time to determine how this would translate to a "AR webpage" placed in the real world. Fortunately, previous research into "spatial AR" (Thomas et al. 2014) helped develop a concept.

In the concept of spatial AR, all AR content in a physical space is anchored above flat surfaces such as tables. Placing AR objects near surfaces like these gives users specific locations and context that helps them understand how the AR object works.

The Room Model merges the concept of spatial AR with the AR Container. Each AR Container has an anchor, a real-world object that the Container is placed near to. The AR Container system takes advantage of the physical affordances of an anchor to size itself approprietly. For example, the surface above a table becomes a cube-shaped area that content can be placed in.

I developed a Photoshop mockup of how this Room Model might work:

Step 1: Anchor Scan

The Room Model assume the existence of an object scanner - a computer vision component capable of recognizing real world objects that would make good anchors. In the first step of the Room Model process, the object scanner activates and begins scanning a user's room.

This initial scan of the room establishes the overall dimensions of the room and establishes an overall area that AR content could be placed in. The overall Room Model functionality would then generate some base HTML representing the room.

Step 2: Generating Anchors

In the next phase of the Room Model setup, the object scanner scans for potential anchor points that can be used as specific locations to place AR content. Within the overall room HTML, the Room Model then generates initial HTML for each anchor point it identifies.

The Room Model has now set up the room for AR content placement. Specific physical objects ripe for AR content have been identified. In addition, free space between physical objects that could also have AR objects placed in them are identified.

AR Containers Extended For The Room Model

Cube-shaped AR containers might not be the best shape to take advantage of some physical object shapes. I therefore developed differently shaped AR Containers strategically designed to cover a wide variety of physical objects:

  • Ceiling Container: Single large plane
  • Poster Container: Smaller Plane
  • Window Container: Double-sided plane
  • Monitor Container: Single-sided plane designed for computer/TV screen coverage
  • Table Container: Volumetric Cube hovering over the surface area of the table
  • Light Source Container: Sphere that orbits the associated light source
  • Free Space Container: Volumetric Cube between physical objects.

Concept Demo

Now that the initial ideation has established the core concepts of the AR Cube and the Room Model, it was time to start moving towards more robust prototypes that moved towards real-world application. As the concept of using physics to actually control the interactions of AR objects had come up within the Room Model, developing a more functional physics-based interaction system was the next major goal.


Pre-implementation Brainstorming - What Strategies can be used for Physics-based Layouts?

Before I starting developing the physics-based interaction system, I needed to have a stronger idea of what the end goal interactions might be. How would a physic-based system control the exact physical locations of AR content in a coherent way? What strategies could a physic-based layout control use? In particular, what strategies should be used when a set of AR content is moved from one room to another? After a round of brainstorming on potential solutions, I settled on 3 initial strategies for AR room layout control via physics - blackhole, location, and anchor-matching. To more fully-flesh out my paper-based brainstorming, I built an AFrame demo for each strategy, using, for the moment, canned animations rather than physics.

Blackhole Strategy

Live Demo Link (Works best in Firefox)

The first physics-layout control strategy, "blackhole" revolves around the concept of a "master" object that's automatically placed in the center of a given room. When AR objects are moved into this room, the blackhole strategy pulls all AR objects to the center of the room. The blackhole strategy is used as a default or "safe" strategy, when there are enough unknowns that other strategies won't work. This strategy guaruntees an initially usable AR object layout, as it pulls all AR objects to a centralized location that will be accesible in most rooms. However, this stategy will never result in an ideal layout that's truly convenient for the user, and therefore should only be used as a fallback solution.

Location Strategy

Live Demo Link (Works best in Firefox)

The second control strategy, "location" looks at how AR objects were positined in the users' previous room, and attempts to place the AR objects in the same spacially-relative position in the new room. This strategy would potentially work well for room tranfers where the rooms are of similar dimensions, but is otherwise the most limited of the three strategies.

Anchor-Matching Strategy

Live Demo Link (Works best in Firefox)

The third strategy, "anchor matching", revolves around the concept of "anchors" that was previously defined in the Room Model. This strategies looks at what physical anchors each piece of AR content was connecting to in the original room. It then attempts to find a matching physical anchor to place the content near (i.e if an AR object was originally placed near a window, this strategy will attempt to find a window in the new room to place that AR object near to). This is the most interesting and potentially useful strategy, as it attempts to take the end-user context behind AR placement into account when switching rooms.

Prototype

Discussion with of the physics-based layouts with my academic stakeholder lead to the decision to create a "physics playground" - a virtual environment in which we could test various physics-based layout strategies to find which ones worked well. I therefore educated myself on the methods AFrame uses to simulate physics, and then built a playground that my stakeholder could use to test various layout controls.

Prototype Development

Playground V1 - Freeform Controls

With this initial version of the playground, I attempted to build a freeform environment for adding simulated AR objects and building layouts. This demo has the following functionality:

  1. The ability to add any number of simulated AR objects, with physics attached. Objects are adding with either a distance constraint (which maintains a constant distance between the object and the black sphere), or with a spring constraint (which adds an invisible spring with tension between the object and the black sphere.)
  2. The ability to turn the physics interactions on and off. This allows the user to turn physics off and manually adjust the position of AR objects. The physics can then be turned on to let physics forces manage the positions of AR objects.
  3. The ability to activate "Blackhole" mode. This functionility uses physics to pull all the AR objects from wherever they are to the black sphere, allows for easy AR layout cleanup. This functionality is a live, working version of the Blackhole animation-based demo discussed earlier.

Playground V2 - Initial Prebuilt Layout

Version 2 added a prebuilt layout to show more discretly what a physics-based layout might look like. In this version, the two AR objects are tied together in a layout by touching them together. The demo detects collision between the two objects and attached them together via a physics "lock" constraint that sticks them together at the collision point. The two AR objects are then manipulated together as one.

Playground V3 - Complex Prebuilt Layout

Live Demo Link (Works best in Firefox)

Version 3 utilizes a more complex layout scenario, with 8 separate AR objects. All physics interactions between the objects are activated by touching them together. The AR objects will attach themselves in a wide variety of ways, and one of the objects is even "squishable" (the object will be reduced in size every time it collides with another object.). This demo allows the user to test a variety of layout configurations without having to utilize the main control panel implemented in version 1.

Final Design Specification

The project now had models for three key components of AR object control: The AR Cube (how to control individual objects), the Room Model (how to control the entire room), and the Physic Model (how to control the direct interactions between objects). How do these models tie together back into the extension of CSS that was the original goal of the project? My final task in this project was to provide a final, merged model that answered this question. I called the final model "Physics CSS", and here's how it works:

Physics CSS Concepts - Core AR Object Properties

The Core AR Object Properties correspond to the AR Cube, in that they define the basic characteristics of an individual AR object. These basic properties are:

  • Mass:
  • Positioning:
  • Anchor Type:
  • Collision Type Property:

Mass

This numeric property corresponds roughly to specificity/importance in standard CSS, but much more granular. The more mass an AR object has, the more important it is in the overall layout. When collision/overall in AR objects occur, a master-servant relationship is established between the two objects to establish hierarchy. The object with the higher mass in the collision is designated the master. In addition, the mass could control other aspects of AR object interactions, such as the look-and-feel styling. An example, an object with a higher mass might reject attempts to overwrite its styling by a lower-mass object.

Positioning

This property is associated with the Room Layout strategies, and are used to determine when an AR object from one room is moved to a new physical space.

  • Blackhole: moves the AR object to a central location within the new room. Considered to be the “safe” option most likely to guarantee a working layout configuration.
  • Location moves the AR object in the new room to a location as relatively (proportionally) close to the object’s positioning in the original room.
  • Anchor Matching: AR object is moved to the physical room object that most closes matches the physical object the AR object was tied to in the original room:

AnchorType

This property incorporates and controls the "anchor" aspects of the Room Model. As part of the this model merger, I brainstormed out several additional anchor types:

  • Ceiling Anchor: Single large plane
  • Poster Anchor: Smaller Plane
  • Window Anchor: Double-sided plane
  • Monitor Anchor: Single-sided plane designed for computer/TV screen coverage
  • Table Anchor: Volumetric Cube hovering over the surface area of the table
  • Light Source: Sphere that orbits the associated light source
  • Free Space: Volumetric Cube between physical objects.

CollisionType

This property roughly corresponds to the border/margin property of the CSS box model, and incorporates the functionality from the Physics Playground. The goal of this property is to determine what behavior the object should display when collision with another object occurs. The possible behaviors are:

  • Lock: Objects stick together on the spot with no distance separation in-between.
  • Distance: Stick Together, but at a pre-defined distance.
  • Repel: Objects forcem themselves away from each other.
  • Sqishiness: Instead of connection, this collision type enables resizing. When the objects collide, items with squishiness get smaller.
  • Nothing: Objects don't build a connection. Physics forces via mass alone determine movement during collusion. Objects are not linked post-collision.
